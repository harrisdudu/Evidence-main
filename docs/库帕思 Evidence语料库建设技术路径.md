# 库帕思 Evidence语料库建设技术路径

# 一、Evidence 语料库与传统语料库在数据处理上的核心区别

Evidence 语料库以 “精准溯源、证据分级、场景适配、智能合成” 为核心目标，服务于各行业专业决策的证据支撑需求，与传统通用语料库在数据处理的底层逻辑、流程、标准上存在本质差异，核心区别体现在数据加工目标、处理流程、质量标准、技术适配、场景落地五大维度，具体如下：

| **对比维度** | **Evidence 语料库** | **传统通用语料库** |
| --- | --- | --- |
| **数据加工目标** | 为各行业决策提供可溯源、高可信度的证据支撑，每一条语料均指向具体业务结论 / 观点的验证，追求 “证据精准、逻辑闭环” | 为通用 NLP 任务（如文本生成、关键词检索）提供语料基础，追求语料的数量、覆盖面，无明确的证据支撑与决策适配目标 |
| **数据处理流程** | 采用 “先筛选再加工、先分级再入库” 流程：先筛选权威行业数据源，再按证据等级分级，后经场景化加工、结构化标注、溯源绑定，最终入库；全流程围绕 “证据” 做闭环设计 | 采用 “先入库再简单清洗” 流程：批量采集各类文本数据，经基础去重、降噪后直接入库，无分级、场景化加工环节，仅做基础格式统一 |
| **数据质量标准** | 1. 来源权威：仅限官方等权威机构，如监管机构、核心期刊等权威来源；<br>2. 证据可溯源：每段语料绑定原始数据源（文档、页码、段落）；<br>3. 等级可区分：按业务证据可信度做分级标注（如监管政策为 S 级、顶刊论文为 A 级）；<br>4. 场景可匹配：语料标注行业细分场景标签（如金融场景下的固收投研、信贷风控） | 1. 基础质量：无明显噪音、重复内容即可；<br>2. 无权威来源要求：可纳入互联网公开数据、非专业自媒体内容；<br>3. 无溯源、分级、场景标注要求，仅做基础的关键词提取 |
| **数据技术适配** | 适配各行业专属 RAG + 多智能体协作体系，语料经**结构化拆解、向量化优化、图谱关联**处理，可直接支撑智能体的问题拆解、证据检索、答案合成；语料设计为可替换组件，支持迭代优化 | 适配通用大模型预训练 / 微调，语料多为纯文本 / 简单结构化格式，仅做基础向量化，无适配多智能体协作的加工设计，迭代仅为简单的语料增删 |
| **数据场景落地** | 深度绑定行业细分场景（例如金融的投研分析、风险控制、合规审核、产品设计、市场研判），语料加工贴合行业业务的决策逻辑，可直接转化为决策证据，无需人工二次整合 | 无明确的行业场景绑定，语料为通用化内容，落地行业场景时需人工筛选、整合，适配性差，易出现 “答非所问” |

简言之，传统语料库是 “通用型数据池”**，追求 “有数据”；而 Evidence 语料库是**“行业专属证据库”，追求 “有高质量、可溯源、能支撑决策的证据”，数据处理的每一个环节均围绕行业专业决策的证据需求展开。

# 二、Evidence 语料库建设步骤

行业 Evidence 语料库建设核心遵循 “权威溯源、场景细分、分级标注、结构化加工、智能适配”**五大原则，围绕行业核心场景，将语料库建设拆解为**8 个标准化步骤 ，从数据源筛选到最终封装成可对接多智能体的知识库，实现全流程标准化、体系化，同时充分结合公司现有 COT 语料、SFT语料 、知识图谱、基石语料库（MD/PDF）资源，减少重复建设，具体步骤如下：

## （一）步骤 1：行业场景精细化拆解与需求梳理

1.  **细化核心场景：**结合公司 Evidence 产品的核心服务对象（例如金融行业的投研人员、风控师、合规专员、业务决策层等），拆解行业核心细分场景，明确各场景的证据需求；核心细分场景示例：权益投研、固收投研、量化分析、信贷风控、反洗钱合规、监管政策解读、产品设计、市场风险研判等。
    
2.  **梳理场景数据需求：**针对各细分场景，梳理具体的数据需求类型（例如金融行业投研场景需 “行业研报、顶刊论文、交易数据”，合规场景需 “监管政策、法规条文、处罚案例”），形成《场景 - Evidence 需求对应表》。
    
3.  **明确数据分级：**明确各场景的数据等级要求（如合规场景优先采用 “监管机构发文（S 级）”，投研场景优先采用 “头部券商研报（A 级）、顶刊论文（A 级）”）。
    

## （二）步骤 2：权威数据源筛选与分类采集

1.  **数据源筛选原则**：摒弃互联网非权威自媒体、无资质平台数据，仅纳入行业**权威、可溯源、更新及时**的数据源（如金融行业可分为官方监管类、专业投研类、学术研究类、市场交易类四大类）；
    
2.  **数据源分类采集**：按《行业场景 - Evidence 需求对应表》，对各类权威数据源做定向采集，同时整合公司内部已有核心数据（例如内部投研报告、风控案例、合规手册），按 “基石语料（原始文档）、加工语料（已做初步处理）” 分类存储，其中基石语料以 MD、PDF 格式保留原始形态，确保溯源性。
    
3.  **数据初检**：对采集的数据源做基础合规性、完整性检查，剔除无效、过期、非权威数据（如已废止的政策、无署名的研报），形成《某行业 Evidence 权威数据源清单》
    
4.  **数据维护更新**：建立数据源更新机制（如监管政策实时同步、研报按日更新）。
    

## （三）步骤 3：业务专家协同的语料初加工与清洗

1.  **团队组建：**组建 “行业业务专家 + NLP 算法工程师” 双团队，开展语料初加工，避免纯技术加工导致的行业业务逻辑偏差；
    
2.  **基础清洗**：对采集的语料做去重、降噪、格式统一处理（如剔除乱码、重复段落、广告信息，将扫描版 PDF 转为可编辑文本）；
    
3.  **业务层面清洗**：由行业业务专家剔除与行业场景无关的内容、修正专业术语错误、补充行业业务逻辑缺失信息，确保语料的**业务准确性**；
    
4.  **过期语料剔除**：针对行业数据时效性强的特点，由业务专家标注语料的有效时间（如政策的实施时间、研报的适用周期），剔除过期失效语料，建立语料时效性标注体系。
    

## （四）步骤 4：行业专属语料标签体系梳理与标注

标签体系是 Evidence 语料库适配场景、支撑智能检索的核心，采用 “多层级、结构化”**标签设计，由行业业务专家主导，算法工程师配合实现标准化标注，标签体系分为**核心标签层**与**扩展标签层 ，全量语料均需完成核心标签标注，扩展标签按需标注，具体如下：

1.  **核心标签层（必标）**：包含**证据等级、细分场景、数据源类型、语料类型、有效时间**5 类标签，每类标签设置标准化子标签（如证据等级：S/ A/ B/ C 级；如金融细分场景：权益投研 / 信贷风控 / 反洗钱合规）；
    
2.  **扩展标签层（选标）**：包含**行业****品类、研究对象、业务逻辑类型、关键词**等标签，适配精细化检索需求（如金融品类：股票 / 债券 / 基金；研究对象：上市公司 / 行业指数 / 监管机构）；
    
3.  **标注执行**：采用 “人工标注 + 模型辅助标注” 模式，业务专家完成标杆语料标注，算法工程师基于标杆语料训练标注模型，实现批量语料的辅助标注，标注完成后由业务专家做抽检审核，标注准确率需≥95%。
    

## （五）步骤 5：语料分级与证据关联

1.  依据行业证据的**可信度、权威性、适用范围**，制定《行业 Evidence 语料等级划分标准》，将语料分为 多个等级，等级直接关联智能体检索的优先级。
    
        **例如，金融行业可以如下划分：**
    
    1.  _S 级：监管机构官方发文（如央行、银保监会、证监会、交易所等）、国家法规条文、国际顶级监管组织文件；_
        
    2.  _A 级：头部券商 / 公募 / 私募投研报告、金融顶刊论文（如《Journal of Finance》）、权威金融数据库（Wind、Choice、Bloomberg）核心数据、行业协会官方指南；_
        
    3.  _B 级：中型金融机构投研报告、核心财经媒体深度分析、金融行业经典案例、公司内部核心投研 / 风控成果；_
        
    4.  _C 级：普通金融机构研究报告、行业周报 / 月报、金融专业书籍章节；_
        
    
2.  **证据关联**：对存在逻辑关联的语料做跨等级、跨类型关联（如 “某监管政策（S 级）” 与 “头部券商对该政策的解读研报（A 级）” 做关联标注），形成证据链，支撑智能体的交叉验证需求。
    

## （六）步骤 6：语料切分与结构化处理（对接步骤三字段模板）

1.  按照本指南第三章制定的**标准化语料切分方式**，对清洗、标注、分级后的语料做精细化切分，将大篇幅原始语料（如研报、政策文件）拆解为 “段落级、句子级” 的最小证据单元，确保每个单元均为独立的、可支撑单一观点的证据；
    
2.  对切分后的最小证据单元做**结构化处理**，按照第三章的**行业 Evidence 语料字段模板**，填充每个单元的标准化字段（如语料 ID、原始来源、证据等级、场景标签、核心内容、关联语料 ID 等）；
    
3.  对结构化后的语料做**格式统一**，分为基石语料结构化文档（绑定 MD/PDF 原始文件）与加工语料结构化文档（COT/SFT 语料），确保可对接公司现有 知识库与 A2A 多智能体框架。
    

## （七）步骤 7：语料向量化与图谱构建

1.  **语料向量化**：采用行业专属大模型做嵌入（Embedding）处理，对结构化后的最小证据单元做向量化，生成语义层面的向量表示，避免通用模型向量化导致的行业语义偏差；向量化后的语料存储至向量数据库，支撑智能体的精准语义检索；
    
2.  **知识图谱构建**：融合公司现有资源，基于结构化语料，提取核心实体（如金融行业的监管机构、上市公司、金融产品、政策名称）、实体关系（如 “政策 - 影响 - 行业”“研报 - 分析 - 上市公司”），构建行业 Evidence 知识图谱，实现语料的**关联化、可视化**检索，支撑智能体的复杂问题拆解与证据链检索；
    
3.  **语料与图谱绑定**：将每个最小证据单元与知识图谱中的实体 / 关系做绑定，实现 “语料 - 图谱 - 原始数据源” 的三重关联。
    

## （八）步骤 8：语料库封装为行业 Evidence 知识库并对接多智能体

1.  **知识库分层封装**：根据语料类型与功能，将语料库封装为**基石语料库、COT 语料库、SFT 语料库、 知识图谱** 几大分层，各分层按行业细分场景做子库划分，形成 “总库 - 子库 - 最小证据单元” 的三级架构，具体要求见本指南第四章；
    
2.  **知识库接口开发**：开发标准化的 API 接口，实现知识库与公司 A2A 多智能体框架的无缝对接，确保问题拆解、证据检索、证据验证、答案合成、溯源管理五大智能体可快速调取、检索、验证语料；
    
3.  **知识库测试与上线**：针对行业各细分场景，设计测试用例（如 “信贷风控中如何识别企业虚假财报”），验证知识库的检索精准度、证据溯源性、场景适配性；测试通过后正式上线，同时建立知识库的日常维护与迭代机制。
    

## （九）补充：语料库持续迭代与维护

建立 “使用 - 反馈 - 优化” 的正向迭代机制，结合智能体的使用数据、业务专家的反馈、数据源的更新，定期对语料库做优化：包括新增权威数据源、剔除过期语料、修正标注错误、补充场景语料、优化知识图谱实体 / 关系，确保语料库的时效性、准确性、场景适配性。

# 三、行业 Evidence 语料切分方式和字段模版

## （一）语料切分核心原则

Evidence 语料切分以 “最小证据单元、独立观点、可溯源、易检索”**为核心原则，摒弃传统的 “按页数、按章节” 粗放式切分，采用**“基于行业业务逻辑的精细化切分”，确保每个切分后的语料单元均可独立支撑一个业务观点 / 结论，且能精准映射至原始数据源的具体位置，同时适配智能体的语义检索与答案合成需求。切分优先级：**句子级切分（优先）> 段落级切分 > 小节级切分**，仅当句子无法形成独立证据时，做段落 / 小节级切分，且切分后的单元不可超过 300 字（如金融投研数据、政策条文等特殊内容可适当放宽，但不超过 500 字）。

## （二）标准化语料切分方式

针对行业不同类型的语料（政策法规、投研报告、学术论文、交易数据、案例分析等），制定专属的标准化切分方式，兼顾业务逻辑与证据单元的独立性，具体如下：

1.  **监管政策 / 法规条文类语料**
    
    1.  切分依据：按**条款、子目**切分，若单条款过长，按**独立规定 / 要求**做句子级切分；
        
    2.  示例：将《证券法》某条 “发行人申请公开发行股票，应当符合下列条件：（一）具备健全且运行良好的组织机构；（二）具有持续经营能力……” 切分为 “条件（一）、条件（二）” 两个独立句子级证据单元；
        
    3.  核心要求：保留条款编号、子目名称，确保与原始政策文件的格式一致，便于溯源。
        
2.  **行业分析报告 / 行业分析类语料**
    
    1.  切分依据：按**核心观点 + 论据**切分，每个核心观点搭配对应的论据形成一个独立证据单元，做句子 / 段落级切分；
        
    2.  示例：将研报中 “2025 年银行业净息差有望企稳回升（核心观点），主要因央行降准降低银行负债成本，同时信贷结构优化提升资产收益率（论据）” 切分为一个独立段落级证据单元；
        
    3.  核心要求：剔除研报中的无关内容（如免责声明、广告、数据图表说明），仅保留 “观点 + 论据” 的核心证据内容。
        
3.  **学术论文 / 顶刊研究类语料**
    
    1.  切分依据：按**研究结论、研究方法、数据结论**切分，每个独立的研究结论 / 数据结论为一个句子级证据单元；
        
    2.  示例：将论文中 “本文通过面板数据模型分析发现，货币政策的传导效率与金融市场成熟度正相关（研究结论），样本期内 A 股市场的传导效率系数为 0.72（数据结论）” 切分为两个独立句子级证据单元；
        
    3.  核心要求：保留论文的作者、发表期刊、研究方法等基础信息，确保证据的学术权威性。
        
4.  **行业案例 / 处罚决定类语料**
    
    1.  切分依据：按**案例核心事实、违规点、处罚结果、监管依据**切分，每个独立模块为一个段落级证据单元；
        
    2.  示例：将反洗钱处罚案例中 “某银行因未按规定识别客户身份（违规点），被银保监会罚款 50 万元（处罚结果），依据《反洗钱法》第三十二条（监管依据）” 切分为三个独立句子级证据单元；
        
    3.  核心要求：保留案例的主体、时间、文号等关键信息，确保案例的可追溯性与参考性。
        
5.  **市场交易 / 数据类语料**
    
    1.  切分依据：按**数据指标、数据时间、数据结论**切分，每个独立的指标数据 + 结论为一个句子级证据单元；
        
    2.  示例：将 Wind 数据中 “2025 年 Q1 沪深 300 指数涨幅为 8.2%（数据指标 + 时间），创近 3 个季度以来新高（数据结论）” 切分为一个独立句子级证据单元；
        
    3.  核心要求：保留数据来源、统计口径、时间区间，确保数据的准确性与可验证性。

# 四、行业 Evidence 知识库组成及建设要求

行业 Evidence 知识库基于公司现有**基石语料库（MD/PDF）、COT 语料库、SFT语料库、 知识图谱 行业知识库**资源，按 “功能分层、场景分库、证据分级”**的原则搭建，整体分为**三大核心分层 ，各分层下按行业细分场景拆分为若干子库，同时包含**七大核心内容模块**，覆盖行业决策所需的全类型证据；各内容模块与三大分层深度绑定，明确其存储形态、建设要求、适配场景，实现 “内容模块化、存储分层化、使用场景化”，支撑全流程需求。

## （一）行业 Evidence 知识库整体架构

1.  **核心分层**：基石语料库（原始层）、COT 语料库（推理层）、SFT 语料库、 知识图谱 行业知识库（结构化 + 图谱层），三层联动，原始层为基础，推理层为核心，结构化 + 图谱层为支撑，实现 “原始溯源 - 逻辑推理 - 结构化检索” 的闭环；
    
2.  **内容模块**：行业政策法规、行业常用名词解释、行业业务逻辑 COT、业务实操说明、期刊论文、行业指南、专业数据库，七大模块覆盖行业核心场景的全类型证据需求；
    
3.  **场景子库**：各分层 / 模块下按行业细分场景拆分子库，实现精细化检索与场景适配。
